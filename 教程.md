# 🌌 异常的冒险之旅：用 GoTryCatch 驯服 Go 的错误洪流

> 费曼曾说："如果你不能简单地解释它，你就没有真正理解它。"这篇教程就是按照这个理念写的——用最简单的话，把 GoTryCatch 的每一个特性讲清楚。

---

## 🎬 故事的开始：为什么需要 GoTryCatch？

想象你在写一个 Web 服务。用户注册时，你需要验证邮箱格式、查询数据库检查用户是否存在、调用第三方 API 发送欢迎邮件。每一个环节都可能出错：

```go
// 传统方式：层层 if err != nil
func registerUser(email string) error {
    if !isValidEmail(email) {
        return fmt.Errorf("invalid email")
    }

    user, err := db.FindUser(email)
    if err != nil {
        return fmt.Errorf("db error: %w", err)
    }
    if user != nil {
        return fmt.Errorf("user exists")
    }

    err = sendWelcomeEmail(email)
    if err != nil {
        return fmt.Errorf("email error: %w", err)
    }

    return nil
}
```

代码像俄罗斯套娃一样，一层套一层。GoTryCatch 提供了另一种思路：**像其他语言一样，用 try-catch 来处理异常**。但它是类型安全的，基于 Go 泛型实现。

```go
// GoTryCatch 方式
func registerUser(email string) {
    gotrycatch.Try(func() {
        validateEmail(email)           // 可能抛 ValidationError
        checkUserExists(email)         // 可能抛 DatabaseError
        sendWelcomeEmail(email)        // 可能抛 NetworkError
    })

    // 每种错误，单独处理
    gotrycatch.Catch[errors.ValidationError](tb, func(err errors.ValidationError) {
        log.Printf("验证失败: %s (字段: %s)", err.Message, err.Field)
    })
    gotrycatch.Catch[errors.DatabaseError](tb, func(err errors.DatabaseError) {
        log.Printf("数据库错误: %s on %s", err.Operation, err.Table)
    })
    gotrycatch.Catch[errors.NetworkError](tb, func(err errors.NetworkError) {
        log.Printf("网络错误: %s", err.URL)
    })
    tb.CatchAny(func(err interface{}) {
        log.Printf("未知错误: %v", err)
    })
    tb.Finally(func() {
        cleanup()  // 无论如何都会执行
    })
}
```

这就是 GoTryCatch 的核心思想：**让错误处理变得结构化、类型安全、易读易维护**。

## 🎯 第一章：Try/Catch/Finally —— 异常处理的三剑客

### Try：开始一段冒险

`Try` 是起点，它执行你传入的函数，并捕获其中发生的任何 panic：

```go
tb := gotrycatch.Try(func() {
    // 这里写可能出错的代码
    // 任何 panic 都会被捕获
    doSomethingRisky()
})
```

`Try` 返回一个 `*TryBlock` 对象，你可以把它想象成一个"异常容器"——它里面装着：有没有出错？出的是什么错？错误处理了吗？

### Catch：精准捕获特定错误

`Catch` 是一个泛型函数，它只捕获你指定类型的错误：

```go
tb = gotrycatch.Catch[errors.ValidationError](tb, func(err errors.ValidationError) {
    // 只有 ValidationError 类型的错误会进入这里
    // err 的类型是确定的，不需要类型断言
    fmt.Printf("字段 %s 验证失败: %s\n", err.Field, err.Message)
})
```

> **为什么是函数而不是方法？**
>
> 你可能想写 `tb.Catch[ErrorType](handler)`，但 Go 语言规定：方法不能有泛型类型参数。所以 `Catch` 必须是一个独立函数，写成 `gotrycatch.Catch[T](tb, handler)`。这是 Go 的限制，不是设计缺陷。

### CatchAny：最后的兜底

`CatchAny` 是一个方法（可以链式调用），它捕获所有未被处理的错误：

```go
tb = tb.CatchAny(func(err interface{}) {
    // 任何未被前面 Catch 捕获的错误都会到这里
    log.Printf("未预期的错误: %v\n", err)
})
```

### Finally：无论成败，都要收尾

`Finally` 保证清理代码一定会执行，无论是否发生了异常：

```go
tb.Finally(func() {
    // 关闭数据库连接
    // 释放文件句柄
    // 清理临时文件
    // 这些代码一定会执行
})
```

> **重要提醒**：如果有错误未被处理（没有匹配的 Catch，也没有 CatchAny），Finally 会先执行清理代码，然后重新抛出异常。

### 完整示例

```go
package main

import (
    "fmt"
    "github.com/linkerlin/gotrycatch"
    "github.com/linkerlin/gotrycatch/errors"
)

func main() {
    tb := gotrycatch.Try(func() {
        gotrycatch.Throw(errors.NewValidationError("email", "邮箱格式不正确", 1001))
    })

    tb = gotrycatch.Catch[errors.ValidationError](tb, func(err errors.ValidationError) {
        fmt.Printf("验证错误 [%d]: 字段 %s - %s\n", err.Code, err.Field, err.Message)
    })

    tb = tb.CatchAny(func(err interface{}) {
        fmt.Printf("其他错误: %v\n", err)
    })

    tb.Finally(func() {
        fmt.Println("清理工作完成")
    })
}
```

## 🔍 第二章：TryBlock 的状态查询 —— 像侦探一样审视错误

有时候，你想知道 TryBlock 里面发生了什么。GoTryCatch 提供了一组查询方法：

### HasError()：有没有出错？

```go
tb := gotrycatch.Try(func() {
    // 正常执行，没有 panic
})

if tb.HasError() {
    fmt.Println("有错误发生")
} else {
    fmt.Println("一切正常")
}
```

### GetError()：拿到错误值

```go
tb := gotrycatch.Try(func() {
    panic("something went wrong")
})

if tb.HasError() {
    err := tb.GetError()  // 返回 interface{}，值是 "something went wrong"
    fmt.Printf("错误值: %v\n", err)
}
```

### GetErrorType()：获取错误类型名

这对于 Agent 或自动化处理特别有用——你可以根据类型名决定如何处理：

```go
tb := gotrycatch.Try(func() {
    gotrycatch.Throw(errors.NewValidationError("name", "不能为空", 1001))
})

errorType := tb.GetErrorType()  // 返回 "errors.ValidationError"

switch errorType {
case "errors.ValidationError":
    // 处理验证错误
case "errors.DatabaseError":
    // 处理数据库错误
}
```

### IsHandled()：错误处理了吗？

```go
tb := gotrycatch.Try(func() {
    panic("oops")
})

fmt.Println(tb.IsHandled())  // false，还没有 Catch 处理

tb = tb.CatchAny(func(err interface{}) {
    fmt.Println("处理了")
})

fmt.Println(tb.IsHandled())  // true，已经被处理
```

### String()：友好的字符串表示

```go
tb := gotrycatch.Try(func() {
    panic("test error")
})

fmt.Println(tb.String())
// 输出: TryBlock{err: string(test error), handled: false}
```

## 🎁 第三章：TryWithResult —— 带返回值的 Try

前面的 `Try` 执行的函数没有返回值。但很多时候，我们需要执行一个函数并获取结果。这时就用 `TryWithResult`：

### 基本用法

```go
tb := gotrycatch.TryWithResult(func() int {
    return 42  // 正常返回
})

result := tb.GetResult()  // 42
fmt.Println(result)
```

如果发生异常：

```go
tb := gotrycatch.TryWithResult(func() int {
    panic("计算失败")
    return 0  // 这行不会执行
})

result := tb.GetResult()  // 0，因为发生了异常
fmt.Println(tb.HasError())  // true
```

### OnSuccess 和 OnError：成功和失败的回调

```go
tb := gotrycatch.TryWithResult(func() int {
    return computeValue()
})

tb.OnSuccess(func(result int) {
    fmt.Printf("成功! 结果是: %d\n", result)
})

tb.OnError(func(err interface{}) {
    fmt.Printf("失败了: %v\n", err)
})
```

`OnSuccess` 和 `OnError` 都返回 `*TryBlockWithResult[T]`，所以可以链式调用：

```go
tb := gotrycatch.TryWithResult(func() int {
    return computeValue()
}).OnSuccess(func(result int) {
    fmt.Println("成功:", result)
}).OnError(func(err interface{}) {
    fmt.Println("失败:", err)
})
```

### OrElse：失败时提供默认值

```go
tb := gotrycatch.TryWithResult(func() int {
    panic("出错了")
    return 0
})

result := tb.OrElse(100)  // 有错误时返回 100
fmt.Println(result)       // 100
```

### OrElseGet：延迟计算默认值

```go
tb := gotrycatch.TryWithResult(func() int {
    panic("出错了")
    return 0
})

result := tb.OrElseGet(func() int {
    // 只有在出错时才会执行这个函数
    return computeDefaultValue()
})
fmt.Println(result)
```

### CatchWithResult：捕获特定类型的异常

和 `Catch` 类似，但用于 `TryWithResult`：

```go
tb := gotrycatch.TryWithResult(func() int {
    gotrycatch.Throw(errors.NewValidationError("value", "无效", 1001))
    return 0
})

tb = gotrycatch.CatchWithResult[int, errors.ValidationError](tb, func(err errors.ValidationError) {
    fmt.Printf("验证错误: %s\n", err.Message)
})
```

### CatchAnyWithResult：捕获任意异常

```go
tb := gotrycatch.CatchAnyWithResult(tb, func(err interface{}) {
    fmt.Printf("捕获到错误: %v\n", err)
})
```

### TryWithResult 的 Finally

`TryWithResult` 的 `Finally` 方法会返回结果值：

```go
tb := gotrycatch.TryWithResult(func() int {
    return 42
})

result := tb.Finally(func() {
    fmt.Println("清理完成")
})

fmt.Println(result)  // 42
```

## 🏷️ 第四章：内置异常类型 —— 七种武器

GoTryCatch 提供了七种内置异常类型，覆盖了开发中最常见的场景。每种类型都包含丰富的上下文信息：文件名、行号、函数名、时间戳和调用堆栈。

### ValidationError —— 数据验证错误

当用户输入不符合要求时使用：

```go
err := errors.NewValidationError("email", "邮箱格式不正确", 1001)
// 字段名: email
// 错误信息: 邮箱格式不正确
// 错误代码: 1001
```

### DatabaseError —— 数据库操作错误

```go
err := errors.NewDatabaseError("SELECT", "users", sqlErr)
// 操作类型: SELECT
// 表名: users
// 原因: sqlErr（底层错误）
```

`DatabaseError` 有一个特殊之处：它实现了 `Unwrap()` 方法，返回底层错误，可以用 `errors.Is()` 和 `errors.As()` 进行错误链查询。

### NetworkError —— 网络请求错误

有两种构造方式：

```go
// HTTP 错误
err := errors.NewNetworkError("https://api.example.com", 500)

// 超时错误
err := errors.NewNetworkTimeoutError("https://api.example.com")
```

通过 `err.Timeout` 可以判断是否是超时：

```go
if netErr.Timeout {
    fmt.Println("请求超时")
} else {
    fmt.Printf("HTTP 错误码: %d\n", netErr.StatusCode)
}
```

### BusinessLogicError —— 业务规则错误

当业务规则被违反时使用：

```go
err := errors.NewBusinessLogicError("age_limit", "用户必须年满18岁")
// 规则名: age_limit
// 详情: 用户必须年满18岁
```

### ConfigError —— 配置错误

```go
err := errors.NewConfigError("database.host", "", "配置项不能为空")
// 配置键: database.host
// 配置值: ""（空）
// 原因: 配置项不能为空
```

### AuthError —— 认证授权错误

```go
err := errors.NewAuthError("login", "user123", "密码错误")
// 操作类型: login
// 用户标识: user123
// 原因: 密码错误
```

### RateLimitError —— 限流错误

```go
err := errors.NewRateLimitError("api", 100, 150, 60)
// 资源名: api
// 限制: 100 次/分钟
// 当前: 150 次
// 重试等待: 60 秒
```

## 📊 第五章：错误的结构化输出 —— 让机器也能读懂错误

所有内置错误类型都实现了 `ToMap()` 和 `ToJSON()` 方法，方便日志记录和 Agent 解析。

### ToMap()：转为字典

```go
err := errors.NewValidationError("email", "格式不正确", 1001)

m := err.ToMap()
// map[string]interface{}{
//     "type": "ValidationError",
//     "field": "email",
//     "message": "格式不正确",
//     "code": 1001,
//     "file": "main.go",
//     "line": 42,
//     "function": "main.validateEmail",
//     "timestamp": "2024-01-15T10:30:00Z",
//     "stack": [...]
// }
```

### ToJSON()：转为 JSON

```go
err := errors.NewValidationError("email", "格式不正确", 1001)

jsonData, _ := err.ToJSON()
fmt.Println(string(jsonData))
// {"type":"ValidationError","field":"email","message":"格式不正确","code":1001,"file":"main.go","line":42,"function":"main.validateEmail","timestamp":"2024-01-15T10:30:00Z","stack":[...]}
```

这对日志系统和 AI Agent 特别友好——结构化的数据比自由文本更容易解析和分析。

### Unwrap() 和 Is()：错误链支持

`DatabaseError` 包含底层错误，可以用 `Unwrap()` 获取：

```go
dbErr := errors.NewDatabaseError("SELECT", "users", sqlErr)

underlying := dbErr.Unwrap()  // 返回 sqlErr
```

`Is()` 方法用于错误匹配：

```go
err1 := errors.NewValidationError("email", "错误1", 1001)
err2 := errors.NewValidationError("name", "错误2", 1001)
err3 := errors.NewValidationError("age", "错误3", 1002)

err1.Is(err2)  // true，因为 Code 相同
err1.Is(err3)  // false，因为 Code 不同
```

## 🐛 第六章：调试模式 —— 当类型匹配不按预期工作时

有时候，你明明抛出了 `ValidationError`，但 `Catch[errors.ValidationError]` 却没有捕获到。这时可以开启调试模式：

```go
gotrycatch.SetDebug(true)

tb := gotrycatch.Try(func() {
    panic("string error")
})

tb = gotrycatch.Catch[errors.ValidationError](tb, func(err errors.ValidationError) {
    fmt.Println("捕获到验证错误")
})

// 控制台输出:
// [gotrycatch] Catch: type string does not match target type errors.ValidationError
```

调试模式会输出类型匹配的详细信息，帮助你找到问题所在。

查询调试状态：

```go
if gotrycatch.IsDebug() {
    fmt.Println("调试模式已开启")
}
```

## ⚡ 第七章：断言辅助函数 —— 让条件检查更简洁

### Assert：条件断言

```go
// 传统写法
if value == "" {
    panic(errors.NewValidationError("value", "不能为空", 1001))
}

// 使用 Assert
gotrycatch.Assert(value != "", errors.NewValidationError("value", "不能为空", 1001))
```

如果条件为 `false`，就抛出指定的错误。

### AssertNoError：错误断言

```go
result, err := someFunction()
if err != nil {
    panic(fmt.Errorf("操作失败: %w", err))
}

// 使用 AssertNoError
result, err := someFunction()
gotrycatch.AssertNoError(err, "操作失败")
```

如果 `err != nil`，就抛出一个包装后的错误。

## 🔗 第八章：完整示例 —— 真实世界的用法

### 示例1：用户注册流程

```go
func registerUser(email, password string) error {
    var dbConn *sql.DB
    var tx *sql.Tx

    tb := gotrycatch.Try(func() {
        // 1. 验证输入
        gotrycatch.Assert(isValidEmail(email),
            errors.NewValidationError("email", "邮箱格式不正确", 1001))
        gotrycatch.Assert(len(password) >= 8,
            errors.NewValidationError("password", "密码至少8位", 1002))

        // 2. 数据库操作
        dbConn = connectDB()
        var err error
        tx, err = dbConn.Begin()
        gotrycatch.AssertNoError(err, "开启事务失败")

        // 3. 检查用户是否存在
        exists, err := checkUserExists(tx, email)
        gotrycatch.AssertNoError(err, "查询用户失败")
        gotrycatch.Assert(!exists,
            errors.NewBusinessLogicError("duplicate_user", "用户已存在"))

        // 4. 创建用户
        err = createUser(tx, email, password)
        gotrycatch.AssertNoError(err, "创建用户失败")

        // 5. 发送欢迎邮件
        err = sendWelcomeEmail(email)
        if err != nil {
            // 邮件失败不影响注册，只记录日志
            log.Printf("发送欢迎邮件失败: %v", err)
        }

        // 6. 提交事务
        gotrycatch.AssertNoError(tx.Commit(), "提交事务失败")
    })

    // 处理各种错误
    tb = gotrycatch.Catch[errors.ValidationError](tb, func(err errors.ValidationError) {
        log.Printf("验证错误 [%d] %s: %s", err.Code, err.Field, err.Message)
    })

    tb = gotrycatch.Catch[errors.DatabaseError](tb, func(err errors.DatabaseError) {
        log.Printf("数据库错误 %s on %s: %v", err.Operation, err.Table, err.Cause)
    })

    tb = gotrycatch.Catch[errors.BusinessLogicError](tb, func(err errors.BusinessLogicError) {
        log.Printf("业务错误 %s: %s", err.Rule, err.Details)
    })

    tb = tb.CatchAny(func(err interface{}) {
        log.Printf("未知错误: %v", err)
    })

    tb.Finally(func() {
        // 清理资源
        if tx != nil {
            tx.Rollback()  // 如果已提交，Rollback 不会报错
        }
        if dbConn != nil {
            dbConn.Close()
        }
    })

    if tb.HasError() {
        return fmt.Errorf("注册失败")
    }
    return nil
}
```

### 示例2：带返回值的计算

```go
func divideAndLog(a, b int) int {
    tb := gotrycatch.TryWithResult(func() int {
        gotrycatch.Assert(b != 0,
            errors.NewValidationError("divisor", "除数不能为0", 1001))
        return a / b
    })

    tb = gotrycatch.CatchWithResult[int, errors.ValidationError](tb, func(err errors.ValidationError) {
        log.Printf("验证错误: %s", err.Message)
    })

    return tb.OrElse(0)  // 出错时返回 0
}
```

## ⚠️ 第十章：注意事项

### 1. Catch 是函数，不是方法

```go
// ❌ 错误
tb.Catch[ErrorType](handler)

// ✅ 正确
gotrycatch.Catch[ErrorType](tb, handler)
```

但 `CatchAny` 和 `Finally` 是方法，可以链式调用：

```go
// ✅ 正确
tb.CatchAny(handler).Finally(cleanup)
```

### 2. 未处理的错误会重新抛出

如果错误没有匹配的 `Catch`，也没有 `CatchAny`，`Finally` 执行后错误会重新 panic。

### 3. Catch 的顺序很重要

把具体的类型放在前面，通用的放在后面：

```go
// ✅ 正确顺序
gotrycatch.Catch[errors.ValidationError](tb, ...)
gotrycatch.Catch[errors.DatabaseError](tb, ...)
tb.CatchAny(...)  // 放最后
```

### 4. 不要滥用异常

GoTryCatch 适合处理"异常情况"，而不是"正常的业务流程"。如果一个函数经常返回错误，用传统的 `error` 返回值可能更合适。
